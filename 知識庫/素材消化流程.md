# 素材消化流程

> 當使用者貼影片、文章、GitHub repo 或外部資料時，先走此素材消化流程。

## 流程總覽

```
接收素材 → 建立分析文件 → 內容摘要 → 專案應用 → 知識庫歸檔 → 後續行動
```

## 詳細步驟

### 1. 接收素材

**輸入來源**：
- 影片（YouTube、教學影片等）
- 文章（部落格、技術文章等）
- GitHub repo
- 外部資料（PDF、文件等）

**處理原則**：
- 不直接改寫原文，先存到 `Clippings/`
- 建立素材分析文件，使用 `Templates/素材分析.md`

### 2. 建立分析文件

**操作步驟**：
1. 複製 `Templates/素材分析.md` 到 `Clippings/`
2. 命名格式：`YYYY-MM-DD 素材標題.md`
3. 填寫素材資訊區塊
4. 標記來源類型與網址

**注意事項**：
- 不儲存 API key、token、密碼
- 不儲存學生姓名
- 只用座號與班級代號

### 3. 內容摘要

**分析重點**：
- **核心重點**：提取最重要的 3-5 個觀點
- **關鍵概念**：列出需要理解的專業術語或概念
- **實作步驟**：如果有程式碼或操作步驟，列出關鍵流程

**輸出格式**：
- 使用條列式，保持簡潔
- 避免過度技術術語
- 確保可執行性

### 4. 專案應用

**評估面向**：
- **可重用組件**：哪些部分可以應用到專案中
- **注意事項**：實作時需要注意的限制或風險
- **待實作項目**：具體的任務項目，使用 `[ ]` 清單

**原則**：
- 只處理本次任務相關範圍
- 優先遵循既有架構
- 不做無關重構

### 5. 知識庫歸檔

**歸檔位置**：
- 消化後的知識存到 `知識庫/`
- 更新 `知識庫/index.md` 索引

**文件要求**：
- 一律加入 frontmatter：`title`、`date`、`type`、`tags`
- 保留來源連結
- 建立相關標籤

### 6. 後續行動

**標準清單**：
- [ ] 消化到知識庫
- [ ] 更新專案文件
- [ ] 建立任務看板項目

**驗證方式**：
- 檢查 frontmatter 是否完整
- 確認內容可執行
- 更新相關文件引用

## 品質檢查

### 交付前檢核
- [ ] 只處理本次任務需要的範圍
- [ ] 有明確成功標準
- [ ] 已驗證關鍵流程
- [ ] 有錯誤處理或失敗回報
- [ ] 避免儲存敏感資料
- [ ] 更新相關文件或工作紀錄

### 大聲失敗
- 有任何東西被跳過，就不能說完全完成
- 有任何測試沒跑，就不能說測試全部通過
- 預設浮現不確定性，不隱藏風險
